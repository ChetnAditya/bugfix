# Deployment Guide

**Stack:** Neon (PostgreSQL) · Render (backend) · Vercel (frontend)

---

## Prerequisites

- GitHub repo pushed and up to date
- `render.yaml` and `vercel.json` present at repo root (already committed)
- Accounts on [neon.tech](https://neon.tech), [render.com](https://render.com), [vercel.com](https://vercel.com)

---

## Step 1 — Commit deployment config

```bash
git add render.yaml vercel.json
git commit -m "chore: add Render + Vercel deployment config"
git push
```

---

## Step 2 — Neon (free PostgreSQL)

1. Go to [neon.tech](https://neon.tech) → sign up / log in
2. Click **New project** → name it `bugfix` → choose the region closest to you
3. Once created, go to **Dashboard → Connection string**
4. Copy the full `postgresql://...` connection string — you will need it in Step 3

> Neon's free tier has no expiry and includes 0.5 GB storage, which is sufficient for this app.

---

## Step 3 — Render (backend)

1. Go to [render.com](https://render.com) → **New → Blueprint**
2. Connect your GitHub account and select the repo
3. Render detects `render.yaml` automatically and pre-fills the service config:
   - **Build command:** `npm install && npx prisma generate && npm run build`
   - **Start command:** `npx prisma migrate deploy && node dist/index.js`
   - **Health check:** `/api/health`
4. Fill in the environment variables marked `sync: false` when prompted:

   | Variable | Value |
   |----------|-------|
   | `DATABASE_URL` | Neon connection string from Step 2 |
   | `CLIENT_ORIGIN` | Leave blank for now — fill after Step 4 |
   | `CLOUDINARY_CLOUD_NAME` | Your Cloudinary name (or leave blank) |
   | `CLOUDINARY_API_KEY` | Your Cloudinary key (or leave blank) |
   | `CLOUDINARY_API_SECRET` | Your Cloudinary secret (or leave blank) |

   > `JWT_SECRET` is auto-generated by Render — no action needed.

5. Click **Apply** — Render builds and deploys the service
6. Wait for the deploy to show **Live**, then copy the service URL:
   ```
   https://bugfix-api.onrender.com
   ```

---

## Step 4 — Vercel (frontend)

1. Go to [vercel.com](https://vercel.com) → **New Project** → import your GitHub repo
2. Vercel detects `vercel.json` automatically — no framework preset or root directory change needed
3. Before clicking Deploy, add one **Environment Variable**:

   | Variable | Value |
   |----------|-------|
   | `VITE_API_URL` | `https://bugfix-api.onrender.com` (Render URL from Step 3) |

4. Click **Deploy**
5. Once live, copy the Vercel URL:
   ```
   https://bugfix-app.vercel.app
   ```

---

## Step 5 — Wire CORS back on Render

1. Go to Render dashboard → select the `bugfix-api` service → **Environment**
2. Set `CLIENT_ORIGIN` to your Vercel URL:
   ```
   https://bugfix-app.vercel.app
   ```
3. Click **Save Changes** — Render triggers a redeploy automatically
4. Once redeployed, open your Vercel URL and verify login works end-to-end

---

## Step 6 — Seed production data (optional)

If you want the test accounts available in production, run the seed script pointing at the Neon database:

```bash
cd bug-fix-app/server
DATABASE_URL="postgresql://..." npm run seed
```

Replace `postgresql://...` with your Neon connection string.

> **Warning:** Only do this on a fresh database. Running seed twice will create duplicate users.

## Environment Variables Reference

### Backend (Render)

| Variable | Description | Example |
|----------|-------------|---------|
| `NODE_ENV` | Runtime environment | `production` |
| `DATABASE_URL` | Neon PostgreSQL connection string | `postgresql://user:pass@host/db?sslmode=require` |
| `JWT_SECRET` | Secret for signing JWT tokens | auto-generated by Render |
| `CLIENT_ORIGIN` | Allowed CORS origin (your Vercel URL) | `https://bugfix-app.vercel.app` |
| `COOKIE_SECURE` | Send cookies over HTTPS only | `true` |
| `COOKIE_DOMAIN` | Cookie domain (leave blank for cross-domain) | `` |
| `CLOUDINARY_CLOUD_NAME` | Cloudinary account name | `mycloud` |
| `CLOUDINARY_API_KEY` | Cloudinary API key | `123456789` |
| `CLOUDINARY_API_SECRET` | Cloudinary API secret | `abc123...` |

### Frontend (Vercel)

| Variable | Description | Example |
|----------|-------------|---------|
| `VITE_API_URL` | Backend API base URL | `https://bugfix-api.onrender.com` |

## Redeployment

| Trigger | What happens |
|---------|-------------|
| Push to `main` | Render and Vercel both redeploy automatically |
| Schema change | `prisma migrate deploy` runs on every Render deploy — migrations apply automatically |
| Env var change | Manual redeploy required on whichever service was updated |

---

## Troubleshooting

**Login fails / CORS error**
- Confirm `CLIENT_ORIGIN` on Render exactly matches the Vercel URL (no trailing slash)
- Confirm `VITE_API_URL` on Vercel exactly matches the Render URL (no trailing slash)

**500 on first deploy**
- Check Render logs — most likely `DATABASE_URL` is missing or incorrect
- Confirm the Neon connection string includes `?sslmode=require`

**Blank page on Vercel**
- All client-side routes are rewritten to `index.html` via `vercel.json` — if routes 404, confirm `vercel.json` is at the repo root

**Render service sleeps (free tier)**
- Render free tier spins down after 15 minutes of inactivity; first request after sleep takes ~30 seconds
- Upgrade to Render Starter ($7/month) to keep the service always-on
